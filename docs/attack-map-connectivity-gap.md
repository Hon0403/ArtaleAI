## 1. 結論

`ATTACK.json` 的 **折線幾何正確**，`BuildHTopology()` 可正常生成 **10 節點、16 邊**。  
但 `ManualEdgeAnchors` **只有 2 條單向 Jump 上台**，沒有任何 **下台邊**。  

因此：玩家在 **`plat_1`（Y=151.3）** 時，`FindPath` **永遠算不到** `plat_0`（Y=162.5，怪物路 x=8.2～233.2）上的任一節點。  
這是 **地圖行為邊未標全**，不是 `PathPlanningTracker` / A* 程式錯誤。

## 5. 程式行為：`SelectRandomPhysicalTarget` 實際做了什麼

檔案：`Core/PathPlanningTracker.cs`（約 769～840 行）

1. `FindNearestNode(playerPos)` → 玩家在 (190.8, **150.0**) 時 → **`n_v_plat_plat_1_1899_1513`**
2. 候選 = 全部平台節點 **9 個**（排除起點）
3. 對 **每個候選** 呼叫 `FindPath` → 約 **9 次 A*** / 每次隨機規劃
4. 其中 **4 個 `plat_0` 候選全部失敗** → 產生 WRN
5. 僅 **4～5 個上層候選成功** → `reachablePaths`
6. 從 `reachablePaths` **隨機選一條** → 永遠是 `plat_1`/`plat_2` 短路

**log 第一條成功規劃（`13:19:32.361`）：**

```
[路徑規劃] 找到路線！n_v_plat_plat_1_1899_1513 -> n_v_plat_plat_1_1936_1513，保留所有原始節點共 2 個
```

→ 距離約 **3.7 px** Walk，**不是**怪物路 174 px 那段。

**34 次「找到路線」終點統計：**

| 終點平台 | 次數 |
|----------|------|
| `plat_1` | 多數（如 1899↔1936、1899↔1851） |
| `plat_2` → `plat_1` | 若干（經 Jump 3 節點） |
| **`plat_0` 任一節點** | **0** |

## 8. 待選方案（之後決策，暫不實作）

### B. 攻擊模式候選過濾（程式）

`PathPlanningTracker.SelectRandomPhysicalTarget`：自動攻擊開啟時只選 `PlatformId == "plat_0"` 或可達 component 內節點。  
**前提：** 玩家仍須能到 `plat_0`（下跳邊或出生在下層）。

### C. P1 降噪（程式）

component 預過濾，避免對 `plat_0` 的 4 次無效 A* 各打一條 WRN（Issue #7 P1）。
